# Notifications

> Last updated: 2026-06-19

Strategies publish `StrategyNotification` events to `INotificationPublisher` whenever a signal fires. A hosted background worker drains a bounded `Channel<StrategyNotification>` and fans each message out to every enabled `INotificationTransport`.

Two transports ship out of the box: **Telegram** (Bot API, JSON over HTTPS) and **Discord** (channel webhook, JSON over HTTPS). Both are independent — enable either, both, or neither.

For the optional AI-generated commentary line that prepends to every notification, see [ai-analyst.md](ai-analyst.md). For the design rationale (channel + dispatcher + enricher pipeline + signal gate), see [architecture.md](architecture.md).

## Telegram setup

1. Open Telegram, talk to [@BotFather](https://t.me/BotFather), `/newbot`, follow the prompts, copy the token (looks like `123456:AA…`).
2. Get the chat ID: message [@userinfobot](https://t.me/userinfobot) for a personal or group chat (it replies with a numeric ID), or use `@channelname` for a public channel.
3. In the terminal, open **Tools → Settings → Notifications**, scroll to the Telegram block:
   - Tick **Enabled**.
   - Paste the bot token.
   - Paste the chat ID.
   - Click **Save**, then **Send test**. A test message should land in Telegram within ~1 second.

## Discord setup

1. In the target Discord channel: *Edit Channel → Integrations → Webhooks → New Webhook → Copy Webhook URL*.
2. In the terminal's Notifications settings tab, scroll to the Discord block:
   - Tick **Enabled**.
   - Paste the webhook URL.
   - Optionally set a username override (empty = use the webhook's default).
   - **Save**, then **Send test**.

## Local LLM commentary (Ollama)

Ollama is a separate enricher path that adds a one-line plain-English commentary to every notification, generated by a local LLM. Free, runs entirely on your machine, no API key.

1. Install Ollama from <https://ollama.ai/download>. Run it.
2. Pull a model: `ollama pull llama3.2` (or `phi3.5`, `mistral`, whatever you prefer).
3. In the Notifications settings tab, scroll to the Ollama block:
   - Tick **Enabled**.
   - Leave endpoint as `http://localhost:11434`.
   - Set the **Model tag** to the one you pulled.
   - **Save**.

The LLM is a workflow tool, not a predictive signal. Each call takes 1–4 seconds — useless for sub-second HFT, but fine for low-frequency signal mode. If Ollama is unreachable or slow, the enricher times out silently and the notification goes out unchanged.

Ollama and AI Analyst are independent enrichers; both can run at once, and they execute in registration order.

## `IncludeIdleSignals`

Both Telegram and Discord blocks have an **Also send idle (unarmed) signals** checkbox.

- Off by default. Strategies fire "idle" notifications during warmup or sub-armed states; with this off, those land in the strategy's own dashboard but never leave the terminal.
- On = idle signals fan out too. Useful while tuning a strategy; usually too noisy for live trading.

The publisher itself doesn't filter — each transport decides for itself. That makes future per-channel routing decisions (e.g. "send all idle signals to Discord, only armed signals to Telegram") trivial.

## Persistence

Settings save to `%LOCALAPPDATA%\DaxAlgo Terminal\notifications.json`. That file is layered onto the host config with `reloadOnChange: true`, so edits in the Settings tab take effect without an app restart. Transports read `IOptionsMonitor<NotificationsOptions>.CurrentValue` on each send.

```json
"Notifications": {
  "QueueCapacity": 256,
  "Telegram": {
    "Enabled": false,
    "BotToken": "",
    "ChatId": "",
    "IncludeIdleSignals": false
  },
  "Discord": {
    "Enabled": false,
    "WebhookUrl": "",
    "Username": "DaxAlgo Terminal",
    "IncludeIdleSignals": false
  }
}
```

## Signal gate

An `ISignalGate` seam sits between the publisher and the transport fanout. It can veto a notification just before send-out with a short human-readable reason for the log.

- The default (`AllowAllSignalGate`) is a no-op.
- The market-regime feature swaps in `RegimeSignalGate` to suppress `Signal`-kind notifications while the composite is below the risk-off threshold. See [market-regime.md](market-regime.md).

The signal still appears in the strategy's own window — only the outbound alert is dropped, with the reason logged.

## Adding a new transport

Add a class implementing `INotificationTransport` in `src/windows/Pipeline/TradingTerminal.Infrastructure/Notifications/<Channel>/` (and the `src/linux` mirror), plus one DI line in `NotificationsServiceCollectionExtensions.AddNotifications`. The dispatcher auto-discovers transports via `IEnumerable<INotificationTransport>`.

Mirror the Telegram or Discord transport for the shape:

- Inject `IOptionsMonitor<NotificationsOptions>` so `IsEnabled` and the channel-specific config are live.
- Use a named `HttpClient` from `IHttpClientFactory`.
- Catch and log errors — never let one failing transport block the others.
- Add a Settings tab block to drive your options. The `notifications.json` writer auto-picks up the new section as long as it's a top-level key under `Notifications`.

See [contributing.md](contributing.md) for the full step-by-step.

## Notification kinds

`NotificationKind` discriminates the message type. The two that ship today:

| Kind | Source | What it carries |
|---|---|---|
| `Signal` | Strategies, on every fired signal. | Strategy id, symbol, direction (`Long` / `Short`), message body. |
| `RegimeChange` | `RegimeRefreshLoop` when the composite band crosses. | Old band, new band, composite score. Optional — set `MarketRegime:NotifyOnRegimeChange` to disable. |

A transport's `IncludeIdleSignals` toggle only affects `Signal`-kind notifications.
