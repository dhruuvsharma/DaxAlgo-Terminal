How a strategy communicates with the backtest engine

One strategy implementation is packaged, verified, loaded into an isolated worker, driven by chronological market events, and connected to simulated execution through a narrow order-router contract. No second backtest version of the strategy is required.

End-to-end architecture

Build once

A headless factory and one IBacktestStrategy contain the canonical trading logic.

Pack and sign

The canonical manifest names the factory and hashes the engine and private dependencies.

Install immutably

The store separates content identity from exact archive and publisher evidence.

Select a run

The host chooses an exact version, trust lane, instrument, parameters, and market input.

Stage the job

The client re-verifies the install and atomically stages only the headless engine closure.

Activate safely

The worker checks every identity and constructs only the manifest-named factory type.

Run events

The engine drives callbacks while orders and fills travel through the simulated router.

Verify the result

The host accepts output only after hashes, trust, closure, and artifact checks succeed.

The process boundary is coarse: market events stay inside the worker. The host sends a request and exact strategy image, receives bounded progress, then validates terminal result files.

The strategy plug-in contract

IStrategyEngineFactory is the activation point, comparable to an Expert Advisor entry point. The bundle manifest names one exact public parameterless implementation, so the worker never scans assemblies or guesses which strategy to run.

Minimal factory and strategy shape

public sealed class ExampleFactory : IStrategyEngineFactory
{
    public StrategyParameterSchema Schema { get; } = new(
        StrategyParameter.Int("lookback", "Lookback", 20, min: 2, max: 500));

    public StrategyDataRequirement DataRequirement => StrategyDataRequirement.L1;

    public IBacktestStrategy Create(Contract contract, StrategyParameters values) =>
        new ExampleStrategy(contract, values.GetLong("lookback"));
}

public sealed class ExampleStrategy(Contract contract, long lookback) : IBacktestStrategy
{
    public Task OnStartAsync(IClock clock, IOrderRouter router, CancellationToken ct) =>
        Task.CompletedTask;

    public Task OnTickAsync(Tick tick, IClock clock, IOrderRouter router, CancellationToken ct)
    {
        // Update indicators; submit or cancel orders only through router.
        return Task.CompletedTask;
    }

    public Task OnOrderEventAsync(OrderEvent evt, CancellationToken ct) =>
        Task.CompletedTask;

    public Task OnEndAsync(IClock clock, IOrderRouter router, CancellationToken ct) =>
        Task.CompletedTask;
}

Why one implementation works

The factory creates the same strategy object from a contract and validated parameter bag. BacktestStrategyKernelAdapter translates it into the engine's canonical IStrategyKernel callbacks. The author does not maintain parallel live and backtest math.

One installed-bundle job

  1. Host

    Chooses the exact bundle, instrument, typed parameters, market input, seed, deadline, and resource limits.

  2. Client → Store

    Calls VerifyInstallation with content root, archive hash, and the current trust policy.

  3. Client

    Copies the canonical manifest and reachable engine/private-dependency closure into a random staging directory, writes request.json, then atomically publishes the job.

  4. Worker

    Validates protocol, host engine hash, limits, deadline, input length/hash, bundle identity, file set, managed metadata, and dependency closure.

  5. Factory

    The worker constructs the exact manifest-named type, reads its schema and data requirement, validates every supplied value, and calls Create.

  6. Engine

    Runs the strategy through chronological market callbacks and a simulated execution loop. Progress is coarse NDJSON; ticks never cross stdout.

  7. Cleanup

    The engine disposes the strategy adapter. Bundle cleanup idempotently disposes it, unloads the collectible context, and the one-shot process exits.

  8. Client → Host

    Accepts success only after the result manifest, report, artifacts, identities, hashes, trust evidence, and closure all match the request.

Runtime communication loop

1 · Market feedProduces the next chronological L1 quote.
2 · Backtest engineAdvances simulated time and applies the quote to the book.
3 · PortfolioRecords quote-triggered fills before callbacks are delivered.
4 · StrategyReceives queued order events, then OnTickAsync.
5 · Order routerReceives place/cancel requests; no broker client is exposed.
6 · FIFO feedbackNew order transitions return before the next market event.
Engine actionPackaged strategy callbackOrdering
Start OnStartAsync(clock, router, ct) Once, before market data
L1 quote OnTickAsync(tick, clock, router, ct) Clock and quote-triggered fills update first; their order events arrive before the tick callback
Trade / depth OnTradeAsync / OnDepthAsync Adapter mappings exist, but the one-shot bundle worker has no trade/depth input schema yet
Completed bar No factory-v1 callback IStrategyKernel.OnBarAsync is a no-op in the current bundle adapter
Order transition OnOrderEventAsync(evt, ct) FIFO; fully drained before the next market event
End OnEndAsync(clock, router, ct) Once; final orders are settled with the last known quote where possible
Current data limits: installed bundles are single-instrument. The worker accepts L1 and Bars requirement flags, but its current synthetic and Parquet feeds produce L1 quotes only. A bundle must not depend on direct bar, trade, or depth delivery yet.

Identity and trust chain

No display name or single hash is enough. One accepted run binds all of these identities:

  1. Publisher, strategy id, version
  2. Canonical content-root SHA-256
  3. Archive SHA-256 and trust evidence
  4. Strategy engine assembly SHA-256
  5. Private dependency closure
  6. Host Backtest.Engine SHA-256

Store paths do not cross the protocol. The worker receives only the canonical manifest and exact headless engine closure. Optional WPF UI, unrelated resources, receipts, and archive bytes are not staged. The worker separately reports the bytes it actually accepted so the client can compare them with the original selection.

A signature is not a sandbox. It proves that a trusted publisher key endorsed the signed content. Strategy code still runs with the user's account permissions. The collectible load context and one-shot Job Object provide lifecycle and fault isolation, not hostile-code containment.

Result acceptance and failure boundaries

Accepted result must match

  • Protocol, job, engine, SDK, and strategy-contract versions.
  • Original request, parameters, and market input identities.
  • Actual host-engine and strategy-engine hashes.
  • Content root, archive hash, and publisher trust evidence.
  • Exact staged assembly closure.
  • Every artifact's relative path, length, and SHA-256.

Report contains

  • Fill-derived accounting and liquidity attribution.
  • Round-trip trades and equity samples.
  • Drawdown and performance metrics.
  • Optional visual recording.
  • It is not a raw fill-event list.

Remaining Backtest Studio wiring

Studio already has native worker run/cancel, bounded progress, and verified report rendering. A clickable installed-strategy workflow still needs:

  1. Bundle-store root and current trust-policy composition.
  2. Active installation discovery with publisher and trust presentation.
  3. Safe schema discovery through an isolated call or future signed manifest field.
  4. Typed parameter editors generated from that schema.
  5. BacktestJobRequest.CreateInstalledBundle with separate host and strategy hashes.
  6. The same activation contract for optimizer, walk-forward, and marketplace UX.

Until then, this page describes the implemented host API and worker route, not a current clickable .daxstrategy picker.
